Introdução

Este relatório documenta a concepção, desenvolvimento e implantação de uma arquitetura de microsserviços voltada para o setor de saúde, simulando um Prontuário Eletrônico do Paciente (PEP). A aplicação foi projetada para obedecer ao padrão FHIR e utiliza comunicação síncrona via gRPC entre os serviços internos, com um API Gateway (Stub) expondo rotas REST para o frontend.

O objetivo central do trabalho é realizar o deploy da aplicação em um cluster Kubernetes, garantindo alta disponibilidade, tolerância a falhas e escalabilidade sob demanda. O relatório aborda não apenas a configuração de infraestrutura, mas também testes exaustivos de carga, validação do Horizontal Pod Autoscaler (HPA) e a implementação de uma camada de observabilidade com Prometheus e Grafana para monitoramento em tempo real.

Montagem do Kubernetes (Modo Cluster)

A montagem do cluster foi realizada em ambiente local utilizando o Minikube para emular um cluster completo. A infraestrutura foi definida de forma declarativa através de manifestos YAML. Os principais desafios enfrentados incluíram a correta orquestração de dependências (garantir que os bancos de dados subissem e ficassem saudáveis antes dos microsserviços) e a configuração das variáveis de ambiente via ConfigMaps e Secrets.

A comunicação interna foi resolvida através do DNS interno do Kubernetes (Services do tipo ClusterIP), enquanto o API Gateway foi exposto externamente via NodePort/LoadBalancer, permitindo ao Frontend React consumir a API unificada.

As Cinco Fases do Projeto

1. Validação Funcional

Foram realizados testes fim-a-fim interagindo com a interface gráfica. Com o frontend configurado para apontar para o Gateway, testamos a autenticação simulando diversos perfis (Médico, Estagiário e Pesquisador). O Gateway extraiu o JWT com sucesso, fez as validações necessárias e encaminhou a requisição via gRPC. A validação atestou que as regras de negócio de acesso (ex: Pesquisador consumindo dados pseudo-anonimizados pelo serviço Data Transform) operaram conforme o esperado.

2. Testes de Carga (Stress Testing)

Para os testes de carga, utilizamos a ferramenta K6. Foram modelados cenários de simulação de tráfego injetando 10, 50, 100, 500 e 1000 requisições simultâneas contra a rota de listagem de pacientes. Observamos que até 100 usuários virtuais o sistema respondeu com p95 < 200ms. No cenário de 500 e 1000 usuários, notou-se degradação de performance e alguns timeouts por esgotamento de recursos no único Pod do Gateway, justificando a necessidade de replicação.

3. Escalabilidade Horizontal

Aplicando o comando kubectl scale deployment api-gateway --replicas=3, a carga foi distribuída entre três Pods. Ao reexecutar o teste com 500 usuários, o sistema recuperou a estabilidade e o tempo de resposta caiu significativamente, demonstrando que o gargalo computacional foi mitigado pela orquestração do Kubernetes aliada a serviços Stateless.

4. Autoscaling (HPA)

Foi configurado o Horizontal Pod Autoscaler limitando o uso de CPU a 70% da capacidade por Pod (Resource requests configurados adequadamente). Ao simular uma rampa agressiva de acessos, o HPA detectou a anomalia através do Metrics Server e instanciou Pods adicionais dinamicamente (de 1 para até 5 réplicas), sustentando a carga de forma autônoma. Após o fim do teste, os Pods foram terminados de forma gradual.

5. Observabilidade

Instrumentamos os microsserviços (como o Datatransform) para expor métricas Prometheus na porta 9103. No Kubernetes, foi feito o deploy do Prometheus para raspar esses dados e do Grafana para visualização. Criamos um dashboard focado no acompanhamento de Throughput (Requisições por Segundo), Latência de respostas gRPC e Taxa de Erros, permitindo visualizar de maneira clara o comportamento da aplicação sob estresse.

Conclusão

A experiência prática comprovou a robustez da arquitetura de microsserviços quando orquestrada adequadamente. O uso do gRPC otimizou a comunicação backend interna, reduzindo overhead em comparação ao REST, crucial em cenários de alta concorrência. As maiores dificuldades envolveram a curva de aprendizado inicial para configuração correta das redes no Kubernetes (Services e Ingress) e o ajuste fino do HPA. Concluímos que a infraestrutura implementada cumpre os requisitos de disponibilidade e escalabilidade.

Anexos

O código fonte completo deste projeto está versionado em nosso repositório no GitHub. Abaixo, destacamos a configuração central utilizada para o escalonamento horizontal autônomo do Gateway: 

# Exemplo de YAML HPA (Horizontal Pod Autoscaler)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-gateway-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-gateway
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
