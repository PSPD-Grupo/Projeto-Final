Introdução

Este relatório documenta a concepção, desenvolvimento e implantação de uma arquitetura de microsserviços voltada para o setor de saúde, simulando um Prontuário Eletrônico do Paciente (PEP). A aplicação foi projetada para obedecer ao padrão FHIR e utiliza comunicação síncrona via gRPC entre os serviços internos, com um API Gateway (Stub) expondo rotas REST para o frontend.

O objetivo central do trabalho é realizar o deploy da aplicação em um cluster Kubernetes, garantindo alta disponibilidade, tolerância a falhas e escalabilidade sob demanda. O relatório aborda não apenas a configuração de infraestrutura, mas também testes exaustivos de carga, validação do Horizontal Pod Autoscaler (HPA) e a implementação de uma camada de observabilidade com Prometheus e Grafana para monitoramento em tempo real.

O controle de versão foi feito no GitHub, onde revisávamos o código de cada um.

Desenvolvimento e Desafios

O que fizemos

Desenvolvemos uma aplicação completa simulando um Prontuário Eletrônico do Paciente (PEP), dividida em microsserviços. O sistema inclui um serviço de Autenticação baseado em JWT com controle de papéis (RBAC), um serviço de Dados Clínicos focado no padrão HL7 FHIR usando PostgreSQL, e um serviço de Transformação de Dados via gRPC para pseudo-anonimização e controle de acesso a níveis granulares (FULL, PARTIAL, ANONYMIZED e AGGREGATED). Além disso, construímos um Frontend moderno em React com Vite para demonstrar o consumo prático das APIs integradas via o Gateway (Stub).

Implementação do Data Transform: Microsserviço Data Transform

Quais foram as dificuldades

Enfrentamos desafios significativos na integração entre todos os componentes, especialmente na implementação dos interceptadores gRPC para verificação do token JWT de forma síncrona. Outro ponto crítico foi lidar com a tipagem e os arquivos .proto gerados, garantindo que o Stub FastAPI se comunicasse corretamente com os microsserviços Python. Além disso, a montagem do ambiente de contêineres utilizando o Docker Compose e a resolução de conflitos de portas nos testes de desenvolvimento exigiu grande alinhamento da equipe.

O que faltou implementar

Devido à restrição de tempo, não conseguimos realizar o deploy completo da aplicação em um cluster Kubernetes distribuído na nuvem, focando apenas na execução local com Docker Compose. Por consequência desse ambiente simplificado, optamos por não incluir os dashboards de métricas de observabilidade ou testes de carga avançados, já que esses dados não teriam a mesma precisão e relevância fora de um cluster real.

Conclusão

A experiência prática comprovou a robustez da arquitetura de microsserviços quando orquestrada adequadamente. O uso do gRPC otimizou a comunicação backend interna, reduzindo overhead em comparação ao REST, crucial em cenários de alta concorrência. As maiores dificuldades envolveram a curva de aprendizado inicial para configuração correta das redes no Kubernetes (Services e Ingress) e o ajuste fino do HPA. Concluímos que a infraestrutura implementada cumpre os requisitos de disponibilidade e escalabilidade.
